# AGENTS.md — AI 协作规则

本仓库使用 Cursor AI agent 协助开发。本文件定义 AI agent 在每次任务**结束**
时的标准汇报格式，**不**影响项目本身的代码规范（见 `realty_app/README.md`
与各子模块 README）。

## 适用范围

- 凡是 agent 通过工具改动文件（新增/修改/删除）并向用户汇报"做完了"时
- 凡是用户说"继续"、"提交"、"push"等收尾指令后
- 凡是一次完整任务结束、准备进入下一轮用户交互前

## 任务结束汇报模板

每次任务结束**必须**输出以下 5 个段落（中文），缺一项视为未完成：

### 1. 修改文件清单

按以下格式列出**所有**改动文件，**不**省略：

```text
modified  path/to/file1.ts           # 一句话说明改动
modified  path/to/file2.vue          # 一句话说明改动
added     path/to/new_file.ts        # 一句话说明新增目的
deleted   path/to/old_file.ts        # 一句话说明删除原因
```

要求：
- 路径使用仓库相对路径（以 `E:\github\application\realty` 为根）
- 每个文件后跟 1 行注释说明**该文件**改了什么
- 不要把多个文件的改动合并成一行

### 2. 修改内容总结

用 1-3 段中文段落总结：
- **这次解决了什么问题** / **完成了什么任务**
- **关键技术决策**（如：为什么选 X 不用 Y）
- **改动覆盖范围**（哪些模块、哪些页面、哪些 API）

不要照抄 commit message 的 body（用户能看到）。重点是让用户**不必打开文件**
也能理解变更意图。

### 3. 方案的优缺点

**必须**诚实列出，至少包含：

| 维度 | 内容 |
|------|------|
| ✅ 优点 | 解决了什么 / 副作用小 / 兼容性好 / 易回退 |
| ⚠️ 缺点 | 引入什么 / 牺牲什么 / 性能影响 / 风险 |
| 🔄 替代方案 | 考虑过但未选的方案 + 不选的理由 |

**禁忌**：
- 只列优点不列缺点（隐式推销）
- 把"未做"的事包装成"优点"（如"未引入新依赖"）
- 用模糊话术（"可能有问题"、"也许更好"）回避明确判断

### 4. 下一步建议

**有**则列出（按优先级排序），**没有**则明确写"暂无后续"。

常见类型：
- 已知 TODO（如本仓库 `changelog/2026-07-12-v0.3.0-优化批次.md` 中"后续可做"）
- 跑验证命令（`npm test` / `npm run type-check` / `node tests/e2e/smoke.mjs`）
- 文档同步（README / DATA_SOURCES.md / changelog）
- 提交与推送（提示 `git add` 范围 / commit message 草稿）
- 重构机会（发现的可以以后再做的小改进）

### 5. 验证状态

如果本次改动包含代码（不是纯文档），必须明确给出以下三项的状态：

```text
- npm run type-check: ✅ 通过 / ❌ 失败（错误数）
- npm test:           ✅ N/N 通过 / ❌ 失败（X 个用例）
- E2E smoke:          ✅ pass / ❌ fail / ⚠️ 未跑（原因）
```

如果某项**未跑**，要说明为什么（时间太长 / dev server 未起 / 不适用）。

## 提交规范联动

完成代码改动后，**提交前**应主动检查（不要求 AI 执行 git 命令，由用户授权）：

- 涉及 `realty_app/` 代码 → 需同步更新 `realty_app/README.md` 底部更新日志
- 版本号变更 → 顶部版本表**新增**一行（**不要修改**历史版本行）
- 详细变更 → `realty_app/changelog/YYYY-MM-DD-vX.Y.Z-变更标题.md`

详细规则见 `realty_app/README.md` 第 261 行起的「提交规范」。

## 反例（不要这样汇报）

❌ "改完了。详见 git log。"
❌ "我做了类型收敛和 a11y 改进。"
❌ "方案很好，没问题。"
❌ "后续可以继续优化。"

✅ 见本文件「任务结束汇报模板」段落。
