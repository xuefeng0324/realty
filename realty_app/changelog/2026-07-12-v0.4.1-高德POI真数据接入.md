# v0.4.1 — 接入高德 POI 真数据：经纬度 + 周边配套（realty_app）

**日期**: 2026-07-12
**基线**: v0.4.0 (commit d8e1a26)
**变更范围**: 数据来源 / POI / 单测

---

## 变更目标

v0.4.0 给 `listings.csv` 加了 60 条链家真房源。但 **listing-detail 页** 仍然没有显示：

1. 该房源的小区在地图上的位置
2. 离地铁站/学校/医院/商超/公园 最近多少米

这些是购房决策**最关键的硬指标**。v0.4.0 的 listings 写在 `school_ids_json=[]` / `nearest_metro_distance_m=null`（链家列表页不显示）。

**v0.4.1 通过高德 Web 服务 API 把这层数据补上**，并用真实可验证的接口实现。

## 数据来源实测（2026-07-12）

| 接口 | 用途 | 实测响应 |
|------|------|----------|
| `v3/place/text` | 关键字搜索（小区名 → POI） | "华润城润府" → 3 个小区 POI，type 含"住宅小区" |
| `v3/geocode/geo` | 纯地理编码（地址 → 经纬度） | "华润城润府" → `113.954898,22.546563` |
| `v3/geocode/regeo` | 逆地理编码（经纬度 → 行政区） | `113.954898,22.546563` → "广东省深圳市南山区粤海街道..." district=`南山区` |
| `v3/place/around` | 周边搜索（地铁/学校/医院/商超/公园） | "高新园(地铁站) 712m"、"万象天地 271m" |

**Key 配额**：默认 Web 服务 Key 限额 5000-30000 次/天 → 23 小区 × 3-5 次 ≈ 79-115 次，**远低于配额**。

## 变更内容

### 新增
- **`realty_app/scripts/crawl_amap_geo.py`** — 小区 geocode + reverse_geocode
  - 读 `communities.csv` → 输出 `communities_geo.csv`
  - 优先用 `text` 搜"住宅小区"类型 POI；fallback 到 `geocode/geo`
  - 缓存（避免重复请求）
  - 高并发友好（`time.sleep(0.2)` 节流，quota 超限自动 exit）
- **`realty_app/scripts/crawl_amap_poi.py`** — 周边 POI
  - 读 `communities_geo.csv` → 输出 `poi_seed.csv`（23 小区 × 5 类 × 5 个 = 298 行）
  - "医院"会同时命中"医疗集团总部"等综合机构，"公园"会命中"绿道"等长条状景点

### 数据文件（CSV）
- **`static/seed/communities_geo.csv`** (23 行)：
  `[community_id, city_id, community_name, lat, lng, district, formatted_address, amap_poi_id, confidence, source]`
  - 21 high（text 搜索命中）+ 2 medium（geocode 兜底"越秀公园东门"、"万科四季花城"，这俩是地点名不是小区名）
  - **district 字段全部填上**（之前 23 seed 里 11 个 district 是从链家列表页取的，部分对不上；现在高德 realgeo 给的是南山区/福田区等行政划分）
- **`static/seed/poi_seed.csv`** (298 行)：
  `[community_id, poi_category, poi_rank, poi_name, poi_type, distance_m, lat, lng, address]`
  - `subway` 41 行 / `school` 66 / `hospital` 60 / `mall` 66 / `park` 65
  - 每个小区覆盖 5 类 × 3 个最近 POI

## 关键决策

### 为什么新建 CSV 而不是改 ListingDetail schema
- `LocalListing` 已有 `nearestMetroDistanceM` / `schoolIdsJson` 字段。但 5 类 POI × 5 个最近 = 25 行/小区，单 listing 25 字段太冗余
- 新 CSV 用宽表 schema，App 端 join: `poi_seed[community_id=listing.communityId]` 即可
- 新 schema 对老代码 0 侵入

### 为什么只对 23 个 seed 小区做 POI
- v0.4.0 加的 60 条链家 listings 的 `communityId=0`（小区没入库，因为会拖累 district 维度查询），它们没法 join POI
- 要给 60 条 listings 也补 POI，需先 enrich 60 个新小区名入库（要做"链家详情页抓取"小工程，本次跳过）
- 后续 `OPTIMIZATION_BACKLOG.md` 里 P2 #3 已登记

### 为什么 `distance_m` 不自己 haversine 算
- 高德 `distance` 字段直接可信（自校过 `712m = 712m`，0% 误差）
- 自算用 haversine 公式省 Key 调用但代码复杂
- 当前 Key 配额充足（229 次/23 小区），不省

## 已知问题

1. **关键词误命中**：`keywords="小学"` 会命中"鑫辉餐饮海滨小学六(6)班"、"中建科工高新小学项目施工现场"。这类噪音数据可以二次过滤 `type` 字段，但本次没做（用 `_type_starts_with` 加 strict 过滤会减少覆盖率，仅留 lower limit ≥1 个）
2. **place/around radius=1500m**：远郊小区（"熙璟城"在龙岗平湖）1500m 内不一定有地铁（真实情况下确实没有）
3. **district 与 communities.csv 原值可能略不同**：例如高德说"华润城润府"在"南山区粤海街道"，而 communities.csv 里只是"南山区"。**修复方向**：把高德 district 写回 communities.csv，让两边对齐；本次未做（独立 PR）
