# v0.3.0 优化批次-3 — catch 迁移 + http/errorMessage 单测

**日期**: 2026-07-12
**类型**: 类型收敛 + 测试覆盖
**版本**: v0.3.0 (同一版本号下第三批优化)

## 改了什么

### 1. catch 块从 `any` 迁移到 `unknown`

6 个页面里 10 处 `catch (e: any) { errorMsg.value = e?.message || String(e) }`
统一迁到 `catch (e) { errorMsg.value = toErrorMessage(e) }`。

| 文件 | 改动处数 |
|------|--------|
| `src/pages/community/community.vue` | 1 |
| `src/pages/dashboard/dashboard.vue` | 3 |
| `src/pages/listing-detail/listing-detail.vue` | 1 |
| `src/pages/listing-filter/listing-filter.vue` | 1 |
| `src/pages/school/school.vue` | 1 |
| `src/pages/settings/settings.vue` | 4 |

每个文件加一行 `import { toErrorMessage } from "../../utils/errorMessage"`。

**为什么不直接 catch (e: unknown)？** TypeScript 5 之后默认 catch 类型就是 `unknown`，写成
`catch (e)` 等价于 `catch (e: unknown)`，更简洁。`toErrorMessage` 是上一批新增的工具函数，
本次把它推到所有页面调用点。

**收益**:
- catch 类型从隐式 `any` 收紧到 `unknown`，与项目"消灭 any"的总目标一致
- 错误处理路径只有一条（`toErrorMessage`），未来扩展（i18n / 上报）只需要改一个地方
- 不再有 `e?.message || String(e)` 这种散落的模式，每个调用点统一一种风格

### 2. 新增 `tests/http.test.ts`（18 用例）

覆盖 `src/api/http.ts` 的 H5 分支：

- `ApiError` 构造、属性、Error 继承
- `buildUrl`（间接触发 via apiGet）：路径拼接、`/` 自动补齐、参数过滤（undefined/null/空串）、
  encodeURIComponent 对 key/value 都生效
- `setApiBaseUrl` / `getApiBaseUrl`：默认值往返、storage trim、空字符串回退
- `apiGet` 200/204/4xx/5xx/JSON 解析失败的 5 个分支
- `apiPost` 200、body 序列化、空 body 不发 body、4xx 抛 ApiError

**Stub 策略**:
- `globalThis.fetch` 用 vi.fn() 替换
- `uni.getStorageSync/setStorageSync` 在 `globalThis.uni` 上注入最小 stub
  （node 环境下没有 uni 全局，必须手动 stub，否则 http.ts import 即抛）

**为什么不测非 H5 分支（`uni.request`）？** 测试跑在 node 环境，uni-app 的
`#ifdef H5` 宏在 vitest 编译时不展开，所以非 H5 分支是死代码。等真接 sqlite/jest-preset-uni
再补。

### 3. 新增 `tests/errorMessage.test.ts`（10 用例）

覆盖 `toErrorMessage` 各种输入：
- null / undefined → 默认 fallback
- 字符串原样返回
- `Error` / `TypeError` 实例 → 取 message
- `{message: "..."}` / `{errMsg: "..."}` 对象
- message 优先于 errMsg
- 空字符串 message / errMsg 退化到下一个字段
- 自定义 fallback 参数
- 数字 / 布尔走 String() 路径

### 4. `vitest.config.ts` 更新注释

`src/api/` 和 `src/utils/` 不再排除（因为有单测了），只是把注释说清楚。
`src/pages/`、`src/store/`、`src/main.ts` 仍排除。

  对 http 的薄包装，单独测 http 已足够覆盖主要风险。
