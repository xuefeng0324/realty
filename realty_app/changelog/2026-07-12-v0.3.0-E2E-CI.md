# v0.3.0 优化批次-6 — E2E CI 化

**日期**: 2026-07-12
**类型**: CI 扩展 / E2E
**版本**: v0.3.0 (同一版本号下第六批优化)

## 改了什么

新增 `.github/workflows/realty-app-e2e.yml`，把 `tests/e2e/smoke.mjs` 接到 GitHub Actions。

### 触发条件

| 触发器 | 是否启用 | 理由 |
|--------|---------|------|
| push 到 main | ❌ | smoke 装 Chromium 慢（首次 2-3min），不应在每个 push 跑 |
| PR 到 main | ✅ | 守 PR 合并前的最后一道 UI 验证 |
| nightly cron | ✅ | 每日 02:00 北京时间（18:00 UTC）回归；错开 `crawl-daily-wangqian` 01:30 UTC |
| manual dispatch | ✅ | 重跑 / 调试用 |

### 步骤设计

```
1. checkout
2. setup-node v4 + npm cache
3. npm ci
4. npm run build:h5   ← production 构建，比 dev 稳定
5. npx playwright install --with-deps chromium  ← 装 Chromium + 系统依赖
6. 起 serve 静态服务器（端口 5173）  ← npx serve，不引入新依赖
7. node tests/e2e/smoke.mjs           ← E2E 主体
8. node tests/e2e/visual-diff.mjs     ← 视觉回归（软失败）
9. 上传 E2E artifacts（14 天）
10. 失败时打印 smoke.json / dist 大小 / serve 状态
```

### 关键设计

**1. 用 build 而非 dev**

dev server (uni 的 HMR + eruda console) 在 CI 环境不稳定，且 dev 模式的 bug 在 prod 不一定出现。
production 构建产物一次定型，更接近真实部署。

**2. 端口 5173（uni dev 默认 5174）**

`serve` 工具的默认端口是 3000，但 uni 项目常用 5173/5174。我们用 5173 保持与 vite 默认对齐。
`E2E_URL=http://127.0.0.1:5173/` 注入 smoke.mjs 即可。

**3. visual-diff 软失败**

`visual-diff.mjs --threshold 0.02` 在 baseline 不存在时会 exit 2。CI 里用 `continue-on-error: true`
让这步失败不阻塞 — baseline 是人工维护的（`--update-baseline`），首次跑或 PR 改 UI 时常常要更新。
**不软失败**的方案是给 baseline 缺失自动创建，但这违背"基线由人工意图控制"的原则。

**4. 并发组隔离**

`concurrency.group: realty_app-e2e` 与 `realty_app-tests` 分开。两个 workflow 可并行；
同 workflow 内 cancel-in-progress。

**5. timeout 12 分钟**

- build:h5 通常 30-60s
- playwright install 首次 1-2min，二次 ~10s
- serve 启动 < 5s
- smoke 跑 30s
- visual-diff < 5s

12 分钟留足 cold-start 缓冲。

## 验证结果

- workflow 文件结构与已有 `realty-app-tests.yml` 风格一致
- 不动 `realty_app/src/` 与 `tests/`，所以本机不需要重新跑单测
- `rhysd/actionlint` 会自动 lint 本文件（CI 上）
- 实际 E2E 触发要等下次 PR 或 nightly
